## Assignment Intent

The rubric (Option B: "Implement a program with threads") specified six
graded requirements. Mapped against what's actually in `river_raid/`,
verified directly against the source rather than assumed:

1. **VPS.** The application was required to run on a cloud VPS. `SERVER_HOST`
   defaults to `127.0.0.1` (`river_raid/server/network/network.py`), and
   nothing else in the codebase references a specific cloud host. There's no
   evidence in this repository that the server was ever actually deployed
   off localhost -- the security review's own notes describe the real
   deployment model as local/loopback. Whether a VPS was used for the
   original grading demo isn't something the code itself can confirm either
   way.
2. **Data / client separate from the VPS.** Confirmed by the client/server
   split itself: `river_raid/client/` runs locally and sends the human
   player's movement/shooting input ("pilot decisions") to `river_raid/server/`
   over the network connection.
3. **Connection -- SSH keys, shared memory on the VPS, computer-controlled
   vs. player-controlled threads.** Partially confirmed, partially not:
   - The three computer-controlled entity threads are confirmed to be
     `EnemyH`, `EnemyJ`, `EnemyB` (`river_raid/shared/entities.py`,
     spawned/moved in `river_raid/server/game/entity_manager.py`) --
     Helicopter, Jet, and Boat, River Raid's classic enemy types. This is
     exactly the rubric's "H J B." Each spawns and moves on its own
     server-side timer/thread, matching "moves for the threads are
     automatically generated by computer."
   - Player A (the human-controlled ship) is confirmed: movement and firing
     come from the client's keyboard input, relayed to the server.
   - "Shared memory located on the VPS" maps to the server-side
     `game_state.py` object that all of a connection's threads read/write --
     this is in-process shared state guarded by the server's own
     synchronization, not literal OS-level shared-memory IPC
     (`multiprocessing.shared_memory` or similar). No code in this
     repository uses OS shared memory directly.
   - **SSH RSA public/private key authentication was required here and is
     NOT what the code currently does.** The client no longer attempts
     key-based auth at all (`CLIENT_KEY_FILENAME`/`CLIENT_KEY_PASSPHRASE`
     are read but unused -- see Known Issues), since the server's
     `check_auth_publickey` (`river_raid/server/network/ssh_server.py`)
     unconditionally returns `AUTH_FAILED` -- key-based auth is disabled
     server-side, full stop. This was a deliberate choice made during this
     repo's security remediation pass, not an oversight: prior to that fix,
     `check_auth_publickey` existed but unconditionally returned
     `AUTH_SUCCESSFUL` for any key (a vulnerability, not a working
     implementation of the rubric's requirement -- it never actually
     verified a key), and the fix replaced both stubbed auth paths with a
     single real check, password-based (`RIVER_RAID_PASSWORD`), rather than
     implementing genuine public-key verification. So even the original,
     pre-fix submission likely never satisfied this requirement for real;
     it just had an auth check that always said yes. The current code is
     secure, but it satisfies the connection via password auth, not the
     RSA-key auth the rubric specifically asked for.
4. **Security -- same SSH RSA key requirement, repeated.** Same finding as
   above: not currently met via keys. The connection is authenticated and
   secure (see Known Issues / Security Findings below), just via a
   different mechanism than the rubric specified.
5. **Documentation and demo.** Out of scope for this repository -- the
   rubric's Word/PPT documentation and video demo were separate submission
   artifacts, not something tracked in the source code itself.
6. **Submission.** Same as above -- a D2L/course-LMS submission process,
   not something this repository can speak to.

In short: the thread-per-enemy-type design (H/J/B) and the client/server
split are solidly implemented and match the rubric. The SSH RSA key
authentication requirement was not met even before the security fix (the
original check was a rubber-stamp, not real key verification), and there's
no evidence the VPS deployment requirement was fulfilled -- this appears to
have stayed a localhost demo throughout.

## Overview

This repository contains **two separate, non-interoperating implementations**
of a clone of the Atari game *River Raid*, exploring different levels of
concurrency and IPC rather than forming one finished game:

| | `standalone_app/` | `river_raid/` |
|---|---|---|
| Architecture | Single process, one background thread | Client/server, multiple background threads server-side, tunneled over a custom SSH layer |
| Concurrency concepts shown | Basic thread + lock | Threading, per-connection isolation, network IPC over SSH, entity pooling |
| Networking | None -- fully local | `paramiko` SSH transport between client and server |
| Attack surface | None (local-only, no network code) | Yes -- see Security Findings below |

Both are scrappy course code, not production software. See
[Known Issues](#known-issues) before assuming either one "just works," and
see [Status](#status) for the overall project state.

A prior security review covered `river_raid/`'s network layer specifically;
those findings are incorporated below. `standalone_app/` was in scope for
that review too and had no findings -- it has no network code and no
externally reachable surface.

### Additional setup for `river_raid/` only

The networked version needs configuration that is not checked into this
repository (no `.env` file and no SSH key files are present, and no secrets
of any kind are committed). To run it, generate these yourself:

1. Generate an RSA key pair for the server to present over SSH:
   ```
   ssh-keygen -t rsa -b 2048 -f server_key -N ""
   ```
2. Create a `.env` file (both client and server read `os.getenv(...)`):

   ```
   # server
   SERVER_HOST=127.0.0.1
   SERVER_PORT=2200
   SERVER_KEY_FILENAME=server_key

   # client
   CLIENT_HOST=127.0.0.1
   CLIENT_PORT=2200
   CLIENT_USERNAME=player
   CLIENT_KEY_FILENAME=server_key
   CLIENT_KEY_PASSPHRASE=

   # shared -- required by both; server rejects all logins if unset
   RIVER_RAID_PASSWORD=change-me
   ```

   A minimal `.env.example` is checked into the repo root as a copyable
   starting point for the shared `RIVER_RAID_PASSWORD` value: `cp
   .env.example .env`, then add the remaining server/client variables shown
   above.

   The server now validates `RIVER_RAID_PASSWORD` against what the client
   sends (see Security Findings below) and fails closed if it isn't set, so
   this value is no longer cosmetic -- pick a real value and use the same
   one in the `.env` that both processes read. The username value is not
   independently checked. **Do not change `SERVER_HOST` off `127.0.0.1` /
   localhost** without also addressing the accepted host-key-trust
   limitation documented below.

3. Run it. Imports in this subproject are package-relative to the
   `river_raid/` directory (there is no top-level `river_raid/__init__.py`),
   so it must be run as a module from inside that directory --
   `python river_raid/client/main.py` directly will fail with import errors.

   ```
   cd river_raid
   python -m server.main    # terminal 1
   python -m client.main    # terminal 2
   ```

### Security Findings

All findings below apply only to `river_raid/` (the client/server/SSH
component). `standalone_app/` is local-only tkinter code with no network
surface and had no findings. No secrets are committed anywhere in this
repo; the setup steps above correctly require generating an SSH key pair
and `.env` file locally rather than shipping them.

Status as of the latest remediation pass: the High, Medium, and one Low
finding below are **Fixed**. One Low finding (host-key trust) is
**Accepted** as a documented limitation rather than fixed -- see its entry
for rationale. The Informational log-rotation item was not addressed
(informational only, not in scope for this pass).

- **[High] No real SSH authentication -- Fixed.**
  `river_raid/server/network/ssh_server.py`, `check_auth_password` and
  `check_auth_publickey` previously returned `paramiko.AUTH_SUCCESSFUL`
  unconditionally, so the SSH layer accepted any username/password/key.
  **Fix applied:** `check_auth_password` now compares the supplied password
  against a `RIVER_RAID_PASSWORD` environment variable and returns
  `AUTH_SUCCESSFUL` only on an exact match, `AUTH_FAILED` otherwise. If
  `RIVER_RAID_PASSWORD` is not set, the server fails closed -- every
  authentication attempt is rejected -- and a warning is printed/logged at
  startup so this isn't silently discovered later. `check_auth_publickey`
  now unconditionally returns `AUTH_FAILED`; there was no existing
  authorized-key store or key-verification mechanism anywhere else in the
  codebase to validate against, so key-based auth is disabled rather than
  faked. `RIVER_RAID_PASSWORD` must be set to the same value in `.env` for
  both server and client -- the server checks it in `check_auth_password`,
  and `river_raid/client/network/network.py` was updated to read it and
  pass it as `password=` to `ssh_client.connect()` (previously the client
  only ever authenticated via `key_filename`, which no longer works now
  that public-key auth is disabled server-side). Without
  `RIVER_RAID_PASSWORD` set, the server refuses all connections and the
  client cannot log in -- this is intentional fail-closed behavior, not a
  bug.
  **Follow-up fix:** the client kept passing `key_filename`/`passphrase` to
  `ssh_client.connect()` alongside `password` even after this change, so it
  still attempted (and always failed) key-based auth before falling back to
  password. That doomed-to-fail attempt also triggered an unrelated
  paramiko bug: `ssh-keygen`'s modern default key format ("OPENSSH PRIVATE
  KEY") gets misidentified as a DSA key during paramiko's key-type
  auto-detection, raising `ValueError: q must be exactly 160, 224, or 256
  bits long` instead of falling through to password auth -- breaking the
  connection entirely rather than just wasting an auth round trip.
  `ClientNetwork.connect()` no longer passes `key_filename`/`passphrase`
  and explicitly sets `look_for_keys=False, allow_agent=False`, so only
  password auth is ever attempted, matching what the server actually
  supports.

- **[Medium] Single-threaded accept loop with unbounded per-connection
  buffer (denial of service) -- Partially fixed (buffer capped and
  one-bad-connection crash fixed; still one connection at a time by
  design).**
  `river_raid/server/network/network.py`, `start_service()` runs a
  synchronous `while True: self.sock.accept()` loop and calls
  `ssh_server.handle_client(channel)` inline, blocking on that one
  connection before the loop can `accept()` again -- a single connected
  client still prevents the server from accepting any other connection.
  Reworking this into a threaded or async/selector-based accept loop was
  explicitly out of scope for this remediation pass (flagged as a larger
  architectural change) and remains unfixed; a comment was added at the
  call site in `network.py` noting this.
  **Fix applied (the buffer-growth half):** `ssh_server.py`'s
  `handle_client()` appended every chunk read from the socket to
  `self.buffer` and only processed it once a `\n` was found, so a client
  that never sent a newline could grow that buffer indefinitely. A
  `MAX_BUFFER_SIZE` (65536 bytes) constant was added; if the buffer exceeds
  this without a newline being found, the connection is closed instead of
  letting the buffer keep growing.
  **Fix applied (a related crash found while wiring up CI):** the entire
  `while True: accept()` loop body was wrapped in a single outer
  `try/except`, so any exception from one connection -- for example a
  plain TCP connect with no SSH banner at all, which raises during
  paramiko's handshake -- propagated out of the loop and killed the whole
  server process, not just that one connection. This is a real
  denial-of-service: one malformed connection attempt (accidental or
  deliberate) permanently took the server down until it was manually
  restarted. Each iteration's connection handling is now wrapped in its
  own `try/except`, so a failed handshake is logged and the loop continues
  accepting new connections instead of crashing.

- **[Low] SSH host key is trusted with no verification -- Accepted
  limitation, not fixed.**
  `river_raid/client/network/network.py`, line 24:
  `self.ssh_client.set_missing_host_key_policy(paramiko.AutoAddPolicy())`.
  This accepts and silently trusts any server host key on first connect,
  which defeats the server-authentication purpose of using SSH as the
  transport (a machine-in-the-middle could impersonate the server
  undetected). **Left as-is deliberately:** switching to
  `paramiko.RejectPolicy()` (or pinning a fingerprint) requires a
  key-distribution mechanism this project doesn't have -- there is no
  provisioning step, no shared `known_hosts` file, and no out-of-band
  channel to hand the client the server's fingerprint ahead of time. Doing
  this properly (not just flipping the policy, which would simply break
  every first connection) needs infrastructure beyond a school project's
  scope. Given the project's actual deployment model -- local/loopback
  demo use, `SERVER_HOST=127.0.0.1` by default, no multi-user or
  non-local hosting -- the practical MITM exposure is minimal, so this is
  recorded as an accepted limitation rather than force-fixed. If this
  project is ever run over an untrusted network, this must be revisited
  first.

- **[Low] Raw exception text returned to the client -- Fixed.**
  `river_raid/server/game/game_manager.py`, `process_message()`, previously:
  ```python
  except Exception as e:
      logging.error(f"Error processing message: {e}")
      return {"status": "error", "message": str(e)}
  ```
  The raw Python exception string was sent back over the channel to the
  connected client, which could leak internal implementation details
  (paths, variable state, stack-adjacent text) to whoever is on the other
  end of the connection. **Fix applied:** the full exception is still
  logged server-side as before, but the client now receives a fixed,
  generic message: `{"status": "error", "message": "Invalid request."}`.

- **[Informational] Client IPs logged without rotation.**
  `river_raid/server/network/network.py` logs each accepted connection's
  address (`logging.info(f"network: Connection from {addr}")`, line 35), and
  `river_raid/server/network/ssh_server.py` configures a `logging.FileHandler`
  to `server.log` (lines 10-17) with no size or time-based rotation. Over a
  long-running server this is an unbounded log file and a minor
  privacy/retention consideration for the IP addresses it accumulates.
  **Fix-it plan:** switch to `logging.handlers.RotatingFileHandler` or
  `TimedRotatingFileHandler`, and document/define a retention policy for
  logged IPs if this is ever run beyond local demos.

- **Checked and confirmed not present:** SQL injection, command injection,
  path traversal, insecure deserialization (message parsing is JSON only,
  via `shared/network_utils.py`), hardcoded secrets/credentials, CORS
  issues, weak/custom cryptography, and file-upload handling. None of these
  apply -- the project has no database, no shell execution, and no web
  frontend.

## Architecture Note: Why SSH?

`river_raid/`'s client and server don't talk over a plain TCP socket
protocol -- the server wraps each incoming TCP connection in a
`paramiko.Transport` and speaks the SSH protocol to it, then tunnels
newline-delimited JSON game messages through an SSH channel. This was
almost certainly chosen to demonstrate OS/networking coursework concepts
(secure channels, transport layers) rather than because a game needs SSH;
a plain socket + JSON protocol would be simpler and is what
`shared/network_utils.py`'s serialize/deserialize helpers would suggest was
the original, simpler design. Given the authentication and host-key
findings above, the current implementation does not actually deliver the
security properties SSH is normally used for -- it currently functions as
an unauthenticated, unverified transport wrapper.
